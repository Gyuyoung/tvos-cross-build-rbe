tvOS cross-compilation on Linux + remoteexec
===

# tvOS Simulator Cross-Compile on Linux (Team Runbook)

## What this does

Builds Chromium tvOS-port targets (`target_os="ios"`, `target_platform="tvos"`,
`target_environment="simulator"`, arm64) on a Linux host by:
- using a Mac-provided AppleTVSimulator SDK bundle locally on Linux,
- overriding the iOS SDK GN args so `sdk_info.py`/`xcodebuild` never run,
- patching a handful of mac-host assumptions in the build config,
- adding the darwin compiler-rt runtimes from Xcode.

Two variants are supported: **Cobalt** (reclient) and upstream **Chromium**
(siso). The Mac-side artifact creation is shared between them.

Everything is driven by these scripts:

| file | runs on | what it does |
|---|---|---|
| `make_artifacts_mac.sh` | Mac | creates the two artifact zips (shared by both variants) |
| `setup_cobalt_linux.sh` | Linux | unpack, compiler-rt install, patches, args.gn, gn gen (Cobalt, reclient) |
| `tvos-cross-cobalt.patch` | — | source patches rebased for Cobalt's older Chromium base |
| `setup_linux.sh` | Linux | unpack, compiler-rt install, patches, args.gn, gn gen (Chromium, siso) |
| `tvos-cross.patch` | — | all source patches, applied by setup_linux.sh |

## Cobalt build

Cobalt (`github.com/youtube/cobalt`) carries an older Chromium base and
uses **reclient**, not siso (`use_siso=false`, `rbe_cfg_dir=//cobalt/reclient_cfgs`,
same `rbe-chromium-untrusted` instance as the Chromium variant below). The
Mac-side artifact creation (`make_artifacts_mac.sh`) is shared — only the
Linux-side setup and patch differ.

Verified working end-to-end 2026-07-15: full `cobalt.app` bundle (Mach-O
arm64) via `//cobalt:cobalt`, ~31k remote actions through reproxy.

### 0) One-time checkout prep

Add `'ios'` to `target_os` in Cobalt's `.gclient` (needed for iOS-only deps
like `//ios/third_party/edo`), then:

```bash
cd "$COBALT_ROOT" && gclient sync
```

Reclient auth is the same as any other Cobalt Linux RBE build (no separate
login step like `siso login`).

### 1) Artifacts

Same as the Chromium flow: run `make_artifacts_mac.sh` on a Mac, copy
`chromium-tvos-toolchain.zip` and `chromium-mac-extra.zip` to the Linux
machine's `~/mac-toolchain/`.

### 2) Set up and generate

```bash
SRC_ROOT="$HOME/projects/cobalt/src" \
ARTIFACTS_DIR="$HOME/mac-toolchain" \
BUILD_DIR="out/tvos-sim" \
BUILD_TYPE=devel \
USE_REMOTEEXEC=true \
./setup_cobalt_linux.sh
```

This unpacks the toolchain, installs darwin compiler-rt into Cobalt's clang,
applies `tvos-cross-cobalt.patch`, writes `args.gn` (imports
`//cobalt/build/configs/tvos-arm64-simulator/args.gn` plus the external-SDK
args and `use_system_xcode=false`), and runs `gn gen --check`. The script is
idempotent — safe to rerun after `gclient sync` (see gotchas below). With a
few thousand GN targets, the `gn gen` step alone can take ~2 minutes; that's
normal, not a hang.

### 3) Build

```bash
autoninja -C out/tvos-sim cobalt
```

`//cobalt:cobalt` is the main tvOS app bundle target.

### How tvos-cross-cobalt.patch differs from tvos-cross.patch

Rebased by hand for Cobalt's older base — same intent as the Chromium patch
(see "What tvos-cross.patch changes" below) but only 5 files: `ios_sdk.gni`,
`compile_xcassets.py`, `csv_password/BUILD.gn`, blink's `scripts.gni`, and
`cobalt/BUILD.gn`. The `user_data_importer` hunk from the Chromium patch
doesn't apply — that code doesn't exist in Cobalt's tree. Cobalt also needs
one extra change not present upstream: `cobalt/BUILD.gn` gates
`//cobalt/browser/tvos/intents:yt_intents` (a Swift AppIntents module) and
`intents_target` behind `host_os == "mac"`, since there's no Apple-target
swiftc on Linux; nothing includes its generated header, so dropping it only
loses Siri/AppIntents integration.

### Cobalt-specific gotchas

- **`gclient sync` wipes the darwin compiler-rt** (same failure mode as
  Chromium's quick-failure-check 1 below) — rerun `setup_cobalt_linux.sh` any
  time after a sync, before building.
- **`cobalt/build/gn.py` requires `-p <platform>`** and its positional arg
  (the out dir) must be two levels deep, e.g. `out/tvos-sim`, not `tvos-sim`
  — only relevant if you regenerate args by hand instead of via the setup
  script.
- Cobalt's base lacks the GN args `xcode_version_verbatim`,
  `use_autogenerated_modules`, and `enable_run_ios_unittests_with_xctest` —
  don't set them even though the Chromium args.gn (and quick-failure-check 6
  below) references `use_autogenerated_modules`.
- System `gperf` (`/usr/bin/gperf`) is used for blink on non-mac hosts
  (patched in `scripts.gni`), same idea as the Chromium variant.

## Chromium build

Verified working: `base`, `net`, `components_unittests`, `content_shell`
(full .app bundle), with siso + RBE (`rbe-chromium-untrusted`).
Compilation runs remotely (~6000 remote steps observed); links run locally.

### 0) One-time checkout prep (Linux)

Add `'ios'` to `target_os` in `$CHROMIUM_ROOT/.gclient`, then:

```bash
cd "$CHROMIUM_ROOT" && gclient sync
```

Log in for remote execution (uses the `rbe_instance` from `.gclient`):

```bash
cd src && siso login
```

### 1) Create artifacts on Mac

Requires Xcode with the tvOS platform installed
(`xcodebuild -downloadPlatform tvOS` if `xcrun --sdk appletvsimulator
--show-sdk-path` fails).

```bash
./make_artifacts_mac.sh
```

Copy `/tmp/chromium-tvos-toolchain.zip` and `/tmp/chromium-mac-extra.zip`
to the Linux machine's `~/mac-toolchain/`.

The toolchain zip contains: `AppleTVSimulator.sdk`, the platform `Library`
(XCTest frameworks), toolchain Swift libs, Xcode `version.plist`, and
`sdk_info.txt` (version numbers consumed by `setup_linux.sh`).

### 2) Set up and generate on Linux

```bash
SRC_ROOT="$HOME/projects/chromium/src" \
ARTIFACTS_DIR="$HOME/mac-toolchain" \
BUILD_DIR="out/tvos_cross" \
USE_REMOTEEXEC=true \
./setup_linux.sh
```

The script is idempotent; rerun it after `gclient sync` (see failure check 1).

### 3) Build

```bash
autoninja -C out/tvos_cross components_unittests
autoninja -C out/tvos_cross content_shell
```

### What tvos-cross.patch changes (and why)

- `build_overrides/build.gni` — relax `assert(current_os != "ios" ||
  use_system_xcode)`: allow iOS targets from a non-mac host.
- `build/config/ios/ios_sdk.gni` — promote `ios_sdk_build` /
  `ios_platform_build` to `declare_args()` (normally only set by
  `sdk_info.py`, which we skip by providing `ios_sdk_path`); relax the
  hermetic-xcode assert.
- `build/config/mac/mac_sdk.gni` + `build/config/apple/sdk_info.py` —
  `mac_hermetic_xcode_path` / flat-layout support (same as the macOS
  runbook; needed because even iOS builds import mac_sdk.gni via
  gn_logs.gni and read `version.plist`).
- `build/config/ios/BUILD.gn` — `rebase_path` the `-isysroot` value when
  host != mac (a source-absolute GN path is passed otherwise).
- `third_party/blink/renderer/build/scripts/scripts.gni` — use the cipd
  gperf on non-mac hosts (upstream runs mac-only `find_sdk.py`).
- `build/config/ios/rules.gni`, `build/config/ios/ios_test_runner_wrapper.gni`
  — don't depend on `//testing/iossim` (mac-only .mm host tool; links an
  empty executable on Linux → `undefined symbol: main`).
- `components/password_manager/services/csv_password/BUILD.gn`,
  `components/password_manager/core/browser/import/BUILD.gn`,
  `components/user_data_importer/utility/BUILD.gn` — swap the iOS Swift
  CSV password parser for the desktop C++ implementation when host != mac
  (no Apple-target swiftc exists on Linux). This was the only Swift in the
  components/content graph.
- `build/toolchain/apple/compile_xcassets.py` — emit stub `Assets.car` /
  partial plist when `xcrun` (actool) is unavailable; asset catalogs are
  not needed to run tests.

### Quick failure checks

1. `error: no such file or directory: ...libclang_rt.tvossim.a` after a
   clang roll / `gclient sync`:
   - the darwin runtimes under
     `third_party/llvm-build/.../lib/clang/*/lib/darwin/` were wiped.
   - rerun `./setup_linux.sh` (step 2 reinstalls them).

2. `need to run siso login`:
   - run `siso login` in `src/` (account with rbe-chromium-untrusted access).

3. `undefined symbol: main` for `clang_x64/iossim`:
   - patch not applied (setup_linux.sh step 3 failed or was skipped).

4. `FileNotFoundError: 'xcrun'` in `compile_xcassets.py` or `swiftc` errors:
   - same as 3 — the patch stubs/bypasses these on non-mac hosts.

5. `directory not found for option -L.../usr/lib/swift/iphonesimulator`:
   - the toolchain bundle lacks the dir; setup_linux.sh creates it, or use
     a bundle produced by the current `make_artifacts_mac.sh`.

6. GN assert `//build/modules/ios-arm64_xcodeNNNN/BUILD.gn does not exist`:
   - args.gn missing `use_autogenerated_modules=false` (regenerate args.gn
     via setup_linux.sh).

## Known limitations

- Anything that links real Swift code (e.g. ios/chrome credential provider)
  cannot be built — no Apple-target swiftc on Linux.
- Built .app bundles are unsigned and asset catalogs are stubs; fine for
  compile checks, not for running on a real simulator (run tests on a Mac).
- `enable_dsyms` must stay false (mac-only dsymutil).
